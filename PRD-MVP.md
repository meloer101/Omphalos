# PRD：AI-native PM 工作台 MVP

> 版本 v0.1 · 2026-07-05 · 上游文件：《产品理念与MVP讨论.md》《Agent架构设计.md》《开源选型表.md》
> 一句话：**4节点心跳（证据→需求→任务→结果）+ 一键追溯 + Cursor式AI侧边栏**，工具即body，图即产品。

---

## 1. Problem Statement

产品团队的工作散落在 Jira/Notion/Figma/Slack/分析工具中，PM 的大量时间花在工具之间做"翻译和搬运"；更致命的是**关系在搬运中丢失**——一个需求为什么做、基于哪些用户证据、上线后验证了没有，最终退化成口头共识、会议印象和聊天片段。团队变大后组织记忆持续损耗：新人无法追溯"当初为什么这么定"，老功能 shipped 之后无人回头验证。现有 AI 工具（Notion AI / Linear AI / Macro）只做环节内生成或无类型关联，**没有产品维护"证据→决策→结果"的有类型因果链**。

## 2. Goals（可衡量的成果）

| # | 目标 | 衡量 |
|---|---|---|
| G1 | **可信追溯**：对任意需求/决策，10 秒内获得"当初为什么"的回答，每句话挂应用内深链、可点击验证 | 回答引用可验证率 100%（无引用的句子不允许出现）；图里没有时明确回答"图里没有记录" |
| G2 | **零成本捕获**：反馈/会议记录/聊天片段丢进来，AI 自动结构化成提议节点+边，用户零手动录入 | AI 提议接受率 ≥70%；高风险因果边错连率 <5% |
| G3 | **验证闭环可见**："shipped 但从未验证"的功能在系统里无所遁形 | 上线需求 30 天内拥有验证/证伪边的比例 ≥50% |
| G4 | **冷启动即有用**：导入团队现有历史后，第一天就能回答第一个追溯问题 | 从导入到首次成功追溯回答 <24 小时 |
| G5 | **降级完整**：AI 关闭时，工作台仍是完整可用的工具 | 侧边栏关闭状态下，收件箱/文档/看板全功能可用 |

## 3. Non-Goals（v1 明确不做）

| 不做 | 理由 |
|---|---|
| 邮件完整客户端 | 重客户端层，独立产品级工程；v1 仅支持邮件转发作为捕获来源（食物不是身体） |
| 原型图/画布 | 画布层，心跳跑通后再做（见架构文件建造分层） |
| 判断 Agent / evidence memo | 图不满时一本正经胡说，会摧毁信任；放最后（决策点：judgment layer 最后做） |
| 自动放权 / 信任刻度 | v1 写死审批规则，只默默记录接受率数据（决策点 10） |
| 通用聊天助手 | 稀释定位、给幻觉留空间；AI 每次露面必须带图的引用（决策点 7） |
| CRDT 实时多人协同编辑 | 工程成本极高；v1 用简单编辑锁+保存提示，架构上为 P2 预留 |
| SaaS 多租户托管 | 商业形态是开源项目，v1 面向自部署 |
| 移动端 | 桌面 Web 优先 |

## 4. User Stories（按优先级）

**PM（核心 persona）**
1. 作为 PM，我想把一堆用户反馈粘贴/转发进收件箱，AI 自动去重、聚类成带证据链的机会点，这样我不用手动整理归类。
2. 作为 PM，我想在需求节点上直接展开写 PRD 正文，这样文档和需求永远不分家。
3. 作为 PM，我想让 AI 把确认后的需求提议拆成任务卡草稿、由我审批落地，这样拆卡不再是体力活。
4. 作为 PM，我想每天用几分钟像审 PR 一样批量审阅"图的变更 diff"，而不是被弹窗逐条打断。
5. 作为 PM，我想在功能上线后把指标结果连回当初的需求，一眼看到验证还是证伪。

**新成员（追溯的第一受益人）**
6. 作为新加入的 PM/工程师，我想问"X 当初为什么这么定"，得到带可点击引用的回答，这样我不用打断三个老同事。

**产品负责人**
7. 作为产品负责人，我想看到哪些已上线的功能从未被验证过，这样立项文化从"产出导向"转向"结果导向"。

**工程师**
8. 作为工程师，我想在任务卡上直接看到它实现的需求和背后的用户证据，不用跑去问 PM"这卡到底为啥做"。

**边缘/负向**
9. 作为 PM，当图里没有答案时，我希望 AI 明确说"图里没有记录"，而不是编一个像样的回答。
10. 作为 PM，我想随时关闭 AI 侧边栏，工作台照常运转。

## 5. Requirements

### P0 — Must-Have（砍掉任何一条，心跳就断）

**R1 图内核**
- Postgres：节点表 + 边表（边带类型）+ pgvector 语义索引
- 边类型封闭枚举 v1：支撑 / 实现 / 验证 / 证伪 / 因为 / 取代 / 重复 / **阻塞**（任务→任务，低风险边，不进信任账本）；非法类型写入直接拒绝
- 边与确认记录 append-only；每条边必须带出处（原始输入、创建 Agent、置信度）
- 验收：
  - [ ] 写入不存在的边类型返回错误
  - [ ] 任意边可查到完整出处链（谁创建、基于什么、谁确认、何时）
  - [ ] 确认记录不可被 UPDATE/DELETE

**R2 四类节点 + 三个原生视图（降级态的完整工具）**
- 反馈收件箱 = 证据节点列表视图；文档页 = 需求节点正文（TipTap/BlockNote 类 MIT 编辑器）；看板 = 任务节点按状态
- 验收：
  - [ ] AI 完全关闭时，三视图可手动完成建节点、连边、编辑全流程（G5）
  - [ ] 节点在任意视图中的修改即时反映到其他视图（同一数据源，无同步）

**R3 捕获 Agent**
- 输入：粘贴文本 / 上传文件 / 邮件转发地址；一个抽取内核+按来源预处理头
- 输出：提议态（虚线）节点与边，永不直接落为已确认
- 验收：
  - Given 粘贴 20 条杂乱用户反馈，When 捕获运行，Then 产出去重聚类后的证据节点 + 与既有需求的"支撑"提议边，全部带出处和置信度
  - [ ] 提议内容可整体拒绝、单条修改后接受

**R4 检索 Agent + 追溯搜索栏（Cmd-K）**
- 两段式：向量定位入口节点 → 沿类型边遍历（数据库查询，零幻觉）→ 生成时仅可引用走到的节点
- 验收：
  - Given 图中存在某决策及其"因为"边，When 问"X 当初为什么"，Then 回答每句挂深链、点击直达节点（G1）
  - Given 图中无相关记录，When 提问，Then 回答"图里没有记录"，不生成推测内容
  - [ ] 回答中不存在无引用的事实性陈述

**R5 PR 式审批**
- 侧边栏内 tab；按边风险分级：低风险（标签/疑似重复）自动生效可撤销，高风险（因为/验证/证伪）必须人显式确认；批量 diff 视图
- 验收：
  - [ ] 高风险边在人确认前对检索 Agent 不可见（追溯永不引用未确认因果边）
  - [ ] 单日 30 条变更可在 5 分钟内完成批量审阅
  - [ ] 每次接受/拒绝按边类型记录（为 v2 放权攒数据）

**R6 AI 侧边栏（Cursor 式）**
- 常驻可开关；永远锚定当前节点为上下文（开任务卡→自动带出证据链与所属需求）；追溯、审批、捕获入口收纳于此
- 验收：
  - [ ] 关闭后无任何 AI 元素残留，工作台全功能可用
  - [ ] 切换页面/节点时侧边栏上下文 <1s 内跟随

**R7 冷启动导入器（第一个"食物"）**
- v1 支持：通用 Markdown/CSV 导入 + **Notion 导入器 + 飞书导入器**（已定，见 OQ1）
- 导入内容经捕获 Agent 熔成原生节点（非镜像、非同步，单向进食）
- 验收：
  - Given 一份导出的历史项目数据，When 导入完成，Then 可成功回答至少一个"当初为什么"级别的追溯问题（G4 演示线）

### P1 — Nice-to-Have（快速跟进）

- **Roadmap 视图**：需求节点按时间/状态的时间轴渲染（纯视图层，低成本）
- **推进 Agent v1**：可配置 playbook（如"证据满 5 条→提议升级为需求"），仅提议
- **指标自动进食**：PostHog/埋点 webhook → 结果节点自动创建，提议"验证/证伪"边（v1 P0 仅支持手动录入结果）
- **Slack / 飞书捕获预处理头**
- **审批准确率面板**：按边类型的接受率统计可视化

### P2 — Future（架构保险，现在不建但不许堵死）

- 画布层（原型图，画布元素可成为图节点）
- 判断 Agent / evidence memo
- 自动放权（基于 R5 攒的数据）
- CRDT 实时协同（解剖 Macro 的 Durable Objects 方案）
- 邮件完整客户端（解剖 Macro 邮件代码）
- schema 版本升级机制承载新节点类型（实验/画像/风险）与团队自定义边类型

## 6. Success Metrics

**Leading（上线后数天~数周）**
| 指标 | 目标 | 测法 |
|---|---|---|
| 捕获提议接受率 | ≥70% | R5 审批日志，按周 |
| 高风险边错连率（被拒率） | <5% | 同上（宁可漏连不可错连的量化） |
| 追溯回答引用点击率 | ≥30% | 前端埋点（点击=用户在验证=信任行为） |
| "图里没有记录"占比 | 记录基线，随图变满应下降 | 检索日志 |
| 日均审批耗时 | <5 分钟 | 审批 tab 停留时长 |

**Lagging（数周~数月）**
| 指标 | 目标 | 测法 |
|---|---|---|
| 每团队每周追溯查询数 | ≥10（北极星候选） | 检索日志 |
| 验证边覆盖率 | 上线需求 30 天内 ≥50% | 图查询 |
| 开源健康度 | stars / 自部署实例 / 外部 PR | GitHub + 匿名遥测（可关） |

**评估节点**：dogfooding 4 周复盘 G1-G5；首个外部设计伙伴团队 8 周复盘 lagging 指标。

## 8. Phasing

| 阶段 | 内容 | 出口标准 |
|---|---|---|
| Phase 0（~2 周） | R1 图内核 + R2 三视图骨架（纯手动可用） | 手动建齐一条"证据→需求→任务→结果"完整链 |
| Phase 1（~3 周） | R3 捕获 + R5 审批 + R6 侧边栏 | 粘贴 20 条反馈→审批落图，全程零手动录入 |
| Phase 2（~3 周） | R4 检索/追溯 + R7 导入器 | **Demo 故事线可演**（见下） |
| Phase 3 | dogfooding：用本工作台管理本产品自身 | 4 周后 leading 指标达标 → 找首个设计伙伴团队 |

### Demo 故事线（Phase 2 出口验收）

1. 导入一份真实的历史项目数据（旧 Notion/飞书导出）；
2. 捕获 Agent 熔图，人在审批 tab 确认因果边（5 分钟）；
3. Cmd-K 提问："结算页当初为什么不做微信支付？"
4. 系统回答：*当时约束是研发排期 + 合规未过，关联这 3 条用户反馈、10 月那次会议、该数据结论*——每句可点击，直达节点；
5. 关闭侧边栏，展示工作台仍是完整的收件箱/文档/看板（降级完整性）。
