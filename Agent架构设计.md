# Agent 架构设计（第一梯队决策记录）

> 前置理念（见《产品理念与MVP讨论.md》）：产品工作的价值不在文档里，在文档之间的关系里。图（graph）是第一等公民，其他一切是它的推论。
> 本文件记录第一梯队三件事的 10 个决策点 + 最终决定，以及讨论后确认的四条关键修正。

---

## 〇、一个先于一切的架构决定：工具是 body，不是外接的手脚

**决定：工作台内的所有工具（看板、文档、反馈收件箱、指标页，终局含 roadmap、PRD、原型图、邮件）原生长在同一个代码库、共用同一个数据库。不走 MCP/API 粘合外部自部署服务的路线。**

**参照物：[Macro](https://macro.com/)（[github.com/macro-inc/macro](https://github.com/macro-inc/macro)）**——"工具即 body"在通用协作领域的完整实现：邮件/消息/文档/任务/通话/CRM 原生一体，AGPLv3 全开源（非 open core），Rust+TypeScript，@链接双向可查询，a16z 领投 $30M（验证了"单一应用+AI-native+开源"赌注可融资）。

**与 Macro 的本质区别（即本产品的差异化）**：Macro 的链接是**无类型**的（@link + 每晚构建的团队记忆），只回答"什么和什么相关"；本产品的边是**有类型**的（支撑/因为/实现/验证/证伪）+ 信任账本，回答"为什么、以及后来验证了吗"。Macro 做上下文，我们做因果与验证闭环。通用定位无法强加封闭边类型枚举——**只有垂直到 PM 工作流，这套语义才定义得出来。垂直即壁垒。**

**终局一致，建造分层**：
| 层 | 内容 | 成本 | 时机 |
|---|---|---|---|
| 视图层 | roadmap、PRD、看板 | 低（需求/任务节点的不同渲染） | MVP |
| 画布层 | 原型图/线框图 | 中（嵌 Excalidraw 式画布，画布元素可成图节点） | 心跳跑通后 |
| 重客户端层 | 邮件 | 高（同步协议/送达率，独立产品级工程） | 最后，届时解剖 Macro 邮件代码 |

邮件的中间态：先作为"食物"（反馈邮件转发进平台 → 捕获 Agent 熔成证据节点，MVP 即可有），完整客户端（body）最后做。

- **买到**：同步问题从根上消失（粘合方案最难的部分就是外部工具与图之间的镜像漂移，它会慢性毒死图的可信度）；所有证据引用都是应用内深链，点击直达图上节点；数据完整性天然保证；PM 永远不需要去别的平台核查。
- **代价与纪律**：从"集成 N 个工具"变成"自己造 N 个工具"，工程量爆炸的经典入口。唯一活路：
  > **不造"N 个工具共享一个数据库"，造"一张图的 N 个视图"。**
  > 看板 = 任务节点按状态的渲染；文档页 = 需求节点的正文展开；反馈收件箱 = 证据节点的列表视图。每个视图只造最小可用层；编辑器/看板等通用组件以 MIT 库形式编进代码（如 TipTap/BlockNote 一类编辑器框架），不从零写。
- **《开源选型表》角色变更**：从"集成对象"变为"解剖对象"——抄 Plane/PostHog 的数据模型思路，不接它们的 API。若直接嵌入开源代码，License 重新相关（本产品自身开源，AGPL 兼容问题不大，但组件优先选 MIT）。
- **保留的例外——外部世界不是 body，是食物**：冷启动导入（Jira/Slack/飞书历史 → 图上原生节点）和"结果"节点的数据源（埋点指标）仍需连接器，但只存在于**进食边界**（单向导入），绝不作为工作界面。
- **降级原则**：AI 失效或侧边栏关闭时，工作台仍是一个完整可用的工具。AI 是增益层，不是承重墙。

## 一、记忆层：图怎么存、怎么读、怎么管

### 决策点 1：图的真身放在哪
**决定：Postgres 装图（节点一张表、边一张表、边带类型）+ pgvector 做语义索引。**
- 否决纯向量库：向量回答"像什么"，图回答"因为什么"，产品卖的是后者；纯向量方案在"当初为什么"问题上是幻觉入口。
- 否决专业图数据库：团队级规模（年几千~几万节点）用不上其强项，多养一个库不值。
- Postgres 事务保证"写边+改状态"原子性（对信任账本关键）；schema 本身是图形状，日后迁移不伤筋动骨。

### 决策点 2：AI 怎么读图
**决定：两段式检索——语义定位入口 → 沿边精确行走 → 生成时只许引用走到的节点。**
1. 向量搜索找入口节点（如《结算页支付方式》）；
2. 沿类型边遍历（"因为"→约束、"支撑"→反馈、→会议记录）——这一步是数据库查询，零幻觉；
3. 模型只组织语言，每句话挂真实节点引用（应用内深链），走不到的一律不说，图里没有就答"图里没有记录"。

### 决策点 3：schema 管多严
**决定：边类型封闭枚举，节点正文自由。**

| 边类型 | 连接 | 示例 |
|---|---|---|
| 支撑 | 证据→需求 | 3 条用户反馈支撑"需要微信支付" |
| 实现 | 任务→需求 | 5 张开发卡实现结算页改版 |
| 验证 / 证伪 | 结果→需求 | 上线后支付转化 +12%，验证当初假设 |
| 因为 | 需求/决策→证据或约束 | 推迟微信支付**因为**合规未过 |
| 取代 / 重复 | 节点→节点 | 新需求取代旧需求；两条反馈重复 |
| 阻塞 | 任务→任务 | 支付接口卡没完成，前端改版卡动不了（低风险边，不进信任账本） |

- 封闭枚举 = 捕获 Agent 每次写入可校验（不存在的边类型直接拒绝），是防止图腐烂的免疫系统。
- 加类型走 schema 版本升级，不许随手加。
- **schema v1 已冻结为 8 种边（2026-07-05）。**

### 决策点 4：历史要不要留
**决定：边和确认记录 append-only，不许 UPDATE 覆盖。** 每条边永久记录：哪个 Agent 基于哪段原始输入创建、谁确认、何时。卖点是"当初为什么"——追溯即时间旅行，历史被覆盖一次，"当初"就没了。

## 二、Agent 架构：怎么分工、怎么协作

### 决策点 5：四个 Agent 按图操作分工，图即总线（事件驱动）
**分工（内核，固定）：**
- **捕获（写）**：乱输入 → 结构化提议节点。最先做、最重要。
- **检索（读）**：查重、挂证据、追溯问答（"当初为什么"就是它接活）。
- **推进（走边）**：沿边提议下一步（"证据够厚了，要不要升级成需求？"），永远只提议。
- **判断（评估）**：evidence memo 等高阶判断。图不满时最容易露怯，**最后做**。

**协作：Agent 之间从不直接通话，全靠往图里写东西协作（事件驱动）。** 捕获写入提议节点→事件→检索被叫醒查重挂边→人确认→事件→推进提议下一步。
- 理由：与"图是第一等公民"同构；每步留痕、审计免费获得；加新 Agent 只需订阅事件，不改旧代码。
- 代价：调试略难于中央编排，四个 Agent 规模下可控。（备选：若需更快出 demo，硬编码流水线先跑，后迁事件驱动。）

### 决策点 6：捕获 Agent 的形态
**决定：一个抽取内核 + 按来源可换的预处理头**（会议/反馈/Slack/旧 Jira），输出统一 schema。图的入口只有一个，可校验点只有一个。

### 决策点 7：不做万能聊天助手
追溯搜索栏背后就是检索 Agent 本人。通用闲聊助手稀释定位且给幻觉留空间。**AI 每次露面都必须带图的引用。**

## 三、自主边界：AI 能自己做多少

### 决策点 8：状态机
**决定：两态——AI 写入一切默认"提议中"（虚线），人确认后"已确认"（实线）。** 硬规则：每条边必须带出处（原始输入、Agent、置信度），无出处的边不允许存在。

### 决策点 9：审批粒度（全产品最危险的体验坑）
风险：每天 50 条待确认 → 用户麻木全选通过 → 信任账本名存实亡。
**决定，三条组合拳：**
1. 按边风险分级：低风险（标签、疑似重复）自动生效可撤销；**高风险——将来会被当事实查的边（因为/验证/证伪）——必须人显式过目**；
2. 审批长成 PR review：攒批 diff，几分钟扫完一天的图变更，不弹窗打断；
3. 调参哲学写死：**宁可漏连，不可错连。** 漏一条边损失便利，错一条因果边失去整个产品的信任。

### 决策点 10：放权演进
**决定：MVP 写死不放权，只默默记录"AI 提议→人接受/拒绝"按边类型的准确率数据。自动放权是 v2 的事。**

## 四、讨论后确认的四条修正（2026-07-05）

1. **分工确认，且定制化边界明确：内核固定，策略开放。**
   - 固定：四 Agent 按图操作分工、两态状态机、出处必填。
   - 开放（公司/用户可定制）：边类型扩展（走 schema 版本升级）、审批风险线（哪些边必须人点头，各团队自己画）、推进 playbook（阈值与动作，如"证据满 5 条提议升级需求"）、捕获预处理头（接飞书还是 Slack）。
   - 一句话：**定制 = 改配置和策略，不是改引擎。**

2. **交互形态收敛为 Cursor 式常驻侧边栏**（可开可关）。
   - 侧边栏永远锚定"当前正在看的节点"作为上下文：开任务卡→自动带出证据链与所属需求；在看板页→推进 Agent 的提议排队于此；PR 式审批收进侧边栏一个 tab。
   - 关闭侧边栏 = AI 整体退场，工作台仍完整可用（降级原则的体现）。

3. **质量评估确认**：涉及证据必须给出链接或指明在项目中的位置——原生架构下全部为应用内深链，点击直达图上节点。

4. **工具层修正**（即第〇节）：不做 claw 式粘合，工具即 body，原生同库；外部系统只在进食边界作为数据来源。

## 五（新增 2026-07-05）、Agent 运行时：记忆、结构与技术选型

### 5.1 记忆架构：图就是记忆，Agent 无状态
不建独立的"agent 记忆系统"（对比 Macro 每晚跑批构建团队记忆，我们的记忆是事务性实时更新的图）。四层记忆全部落在已有设施上：

| 记忆类型 | 落点 |
|---|---|
| 长期共享记忆 | 图本身（决策/证据/因果链） |
| 语义记忆 | pgvector 索引 |
| 校准记忆 | R5 审批日志（AI 提议被接受/拒绝的记录，v2 放权的原料） |
| 会话记忆 | 侧边栏对话（锚定当前节点，短命；有价值结论可被捕获 Agent 提议熔入图——对话也是食物） |

核心机制：**上下文装配函数**——Agent 被唤醒时拿锚点节点查 k 跳邻域+出处链装配 prompt。记忆是每次查出来的，不是存在 Agent 里的。Agent = 无状态 worker，挂了重启零损失（降级原则的又一体现）。

**Project 分割（2026-07-05 定）：物理不分，逻辑分——"分的是视野，不分的是土壤"。**
- 一张图，project 为节点一等属性（scope 字段），不做物理分库/分图；
- 默认局部：上下文装配与追溯检索默认限定当前 project（不串味）；
- 允许跨界边：边可跨 project（低频高价值：共享合规约束、跨产品线同源反馈、A 团队证伪过 B 团队正要做的事），可要求更高确认级别；
- 升级式检索：局部无结果时自动升级查全局——"本项目没有记录，但 B 项目去年有相关证伪结论"是砌墙方案永远给不了的时刻。

### 5.2 Agent 结构：4 个内核，统一五段流水线
```
图事件触发 → 上下文装配（查图） → LLM 调用（结构化输出） → zod 校验（8 种边枚举） → 写提议态节点/边
```
**澄清（2026-07-05）：代码层面就是一个 agent 运行时，跑四套配置。"四个 Agent"分的不是程序，是四份合同（contract）**——一个演员演四个角色，剧本和戏路分开写：

| 合同 | 触发 | 读图 | 写图 | 模型 |
|---|---|---|---|---|
| 捕获 | 新输入到达 | 局部 | 只能写提议态 | 便宜快 |
| 检索 | 用户提问/查重事件 | ✅ | ❌（**数据库凭证物理只读**——安全靠构造，不靠模型自觉） | 最强（严谨引用） |
| 推进 | 确认事件/定时巡图 | ✅ | 只能写提议态 | 中 |
| 判断 | 最后做 | ✅ | ❌ 只产出 memo | 最强 |

合同的意义：权限隔离、按角色评估准确率、按角色配模型。
- 重活（导入 5000 条历史）= 同一运行时多 worker 实例并行（任务队列问题，不是 agent 架构问题）。

### 5.3 框架选型：不用 LangChain
- 理由：我们的 Agent 是窄而结构化的工作流，"下一步做什么"由事件驱动编排层的确定性代码决定，不是 LLM 自主循环——重框架的核心价值用不上，抽象税照付。
- **采用：Vercel AI SDK**（TS 原生契合；`generateObject` 直出 zod 校验的结构化输出；provider 字符串换模型，含本地模型选项以满足 OQ3 采购门槛；流式内建。是库不是框架，不接管架构）。
- **编排层自建且薄**：Postgres LISTEN/NOTIFY 出事件 + pg-boss（Postgres 上的任务队列，仍是单库）+ 数百行编排代码。
- 运行时全家当：AI SDK + zod + pg-boss + 自写编排。

### 5.4 MCP / Skills / Subagent 的位置
- **MCP as server（P1-P2 亮点）**：把图暴露为 MCP server，用户在 Cursor/Claude Code 里直查自己产品的图（"这个函数对应的需求当初为什么做？"）——开源+开发者团队定位的传播卖点。
- **MCP as client / 外部工具（2026-07-05 放宽）**：区分"状态"与"能力"——不镜像外部工具的**状态**（body 原则），但随意调用外部世界的**能力**（感官）：Tavily 搜索、网页抓取、数据补全等可被各 Agent 自由调用。激活"竞品情报"机会点（选型表中的空白）：推进/捕获 Agent 可执行"调研竞品 X 定价"→搜索→熔成带 URL 出处的证据节点。不变量：**凡成为记录的必须落图带出处**；搜完即走的临时信息可不落图，但一旦支撑了决策就必须变成节点。
- **Skills = 策略层的落地形态**：声明式文件（markdown/yaml）承载推进 playbook、审批风险线、捕获预处理头配置——团队定制不碰引擎代码。
- **Subagent**：日常并行=任务队列多 worker，不引 subagent 概念。唯一未来可能需要真·多步 agent loop 的是判断 Agent（多轮查图推理生成 evidence memo），届时候选 Claude Agent SDK，现在不引。
