# AI-native PM 工作台 —— 核心理念与 MVP 讨论

## 一、核心理念（已锁定）

> **产品工作的价值不在文档里，在文档之间的关系里。我们把这层关系变成第一等公民（graph），其他一切都是它的推论。**

这是整个产品的世界观。后续所有功能，凡是能从这句话推导出来的，就是核心；推不出来的，就是杂音，应当砍掉。

### 十个表面优势 → 收敛为一个地基

GPT 提出的十个优势（上下文聚合、artifact-native、决策可追踪、workflow 编排、判断辅助、验证闭环、组织能力补全、产品语义层、人机协作界面、开放自托管）看似并列，实际只有一条承重：**第八条——产品语义层（product ontology / graph）**。

其余九条都是它的下游表现：
- artifact-native = 图里的**节点**
- decision log = 图里的一类节点（Decision）+ 它的边
- 验证闭环 = Hypothesis → Metric → Learning 这条边被自动闭合
- workflow 编排 = 沿着图的边自动推进
- 判断辅助 / evidence memo = 在图上做推理
- embedded agent = 图节点旁边的交互入口

护城河会随使用变深，因为图是团队几个月产品史的累积，别人重装一遍 AI 也抄不走。

## 二、目标用户：公司内部产品团队（已确认）

选择理由：内部团队人多、历史长、扯皮多，"组织记忆"和"追溯检索能力"才有发力的地方。一人公司不需要"当初为什么这么决定"，因为决定者只有他自己。

**这个选择反过来改变了冷启动策略和第一个楔子的选择**（见下）。

## 三、GPT 未讲清楚的关键问题（大白话版）

### 问题一：新用户第一天，你的产品是个"失忆的人"

图的威力来自"记得来龙去脉"，但一个团队刚开始用你的产品时，图里啥都没有。这时候被问"当初为什么这么定"，答不上来——第一天甚至不如 Linear（好歹能建卡干活）。

**但选了"内部团队"，这个问题反而有解**：团队的历史已经躺在他们现有的 Jira / Notion / Slack / 飞书里了。不需要用户从零攒，产品第一步就该去把这些老数据吸进来，自动补一张过去的图。第一天就能回答"当初为啥不做微信支付"——因为老系统里的讨论和决定被吃进来连好了。这正是「内部团队 + 追溯检索」定位能吃到的第一个红利，也是选这个用户群体后独有的机会。

### 问题二：如果录入靠人手动填，图永远是空的

图要满，前提是干活时信息真的"进"到结构里。但如果"记录"是额外的活（干完活还要回来再填一遍），没人会填，图永远残缺——所有知识库类产品都死在这。

**解法**：AI 的第一份工作不是"给建议"，是"替人干录入这件脏活"。开完会甩一段记录进来、丢一堆反馈进来，AI 自动结构化，用户一个字不用敲。结构化产物必须是干活的**副产品**，不能是**额外的作业**。

### 问题三：先造能跳动一次的最小心脏，不要一上来就造 11 种零件的庞然大物

GPT 列了 11 类对象（User Problem、Opportunity、Hypothesis、Feature、Experiment、Metric、Persona、Journey、Decision、Risk、Release、Learning）。全做会累死在半路。

**最小闭环只需要 4 个节点**：**证据（Evidence）→ 需求（Feature/Opportunity）→ 任务（Task）→ 结果（Metric/Learning）**。

翻译：一条用户反馈（证据）→ 变成一个要做的功能（需求）→ 拆成开发任务（任务）→ 上线后指标证明它有用还是没用（结果）。这四个节点串起来跑一遍，就是整个宏大愿景"跳动一次的完整心跳"。其余 7 类对象，等这颗心脏有人用了再逐步长出来。

### 问题四：第一刀下在哪，"AI 拍板判断"必须放到最后

- **"AI 帮你判断该不该做、自动写评审备忘录"这类 judgment layer 功能最酷，但必须最后做。** 图还没满的时候，它就是一本正经地瞎掰，用户一眼看穿，直接失去对整个产品的信任。

- **第一刀该下在"追溯"上。** 原本倾向"反馈 → 需求"作为第一个楔子，但确认目标用户是内部团队、核心卖点是"追溯和检索能力"后，改为：

> **"这事儿当初为啥这么定？"—— 一键问出来。**
>
> 团队新来一个人，或三个月后有人质疑，问一句"结算页当初为什么不做微信支付"，产品直接答：*当时的约束是研发排期紧 + 合规还没过，关联了这 3 条用户反馈、那次 10 月的会、这个数据结论，所以推迟了。*
>
> 这个能力 Notion AI、Linear AI 永远给不了——它们只会"生成新东西"，不会"记得旧决定为什么这么做"。这正好 100% 命中「内部团队 + 追溯检索」的定位，也是空图情况下（问题一的解法）第一天就能秀出来的能力。

## 四、下一步

把「证据 → 需求 → 任务 → 结果」这颗最小心脏 + "一键追溯"这个第一楔子，画成 MVP 的具体形态：
- 长什么样（信息架构 / 交互形态）
- AI 在哪几个环节出现（embedded agent，而非纯 chat-first）
- 第一个 demo 怎么演（老系统历史数据导入 → 追溯问答 的完整故事线）
