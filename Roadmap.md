## Phase 0：图内核 + 三视图骨架（PRD R1、R2）

**目标**：AI 一行代码都没有，但工作台已经是一个完整可用的工具（降级态即出生态）。

### 0.1 工程底座 —— ✅ 完成
- [x] 仓库初始化：TS 全栈（Next.js 16 + Turbopack）、单应用起步（非 monorepo）、AGPL-3.0
- [x] Supabase 本地栈：Postgres + pgvector，`supabase start` 一条命令起本地环境
- [x] CI：GitHub Actions，lint + typecheck + test + build（`pgvector/pgvector:pg17` 服务容器）

### 0.2 图内核（R1）—— ✅ 完成
- [x] 节点表：4 类型枚举（证据/需求/任务/结果）+ project scope 字段 + JSONB 正文 + 状态位（提议中/已确认）+ **新增 `board_status` 列**（看板工作流状态，与信任账本状态正交，见 Phase0-开工计划.md）
- [x] 边表：8 类型枚举（支撑/实现/验证/证伪/因为/取代/重复/阻塞）+ 风险级别标注 + 状态位（risk 由 DB 触发器权威派生）
- [x] 出处表：每条边关联（原始输入引用、创建者、置信度）——无出处的边写入被拒（deferred constraint trigger）
- [x] 确认记录 append-only：数据库触发器禁止 UPDATE/DELETE；已确认节点/边不可删除或改关键字段
- [x] zod schema 与数据库枚举单一来源（防漂移）
- [x] 验收测试：11 个 vitest 用例全绿，覆盖非法边类型/无出处边拒绝/append-only/出处链/CRUD

### 0.3 四类节点的最小 surface（R2）—— ✅ 完成
> **贯穿原则（2026-07-05）：节点早，编辑器晚。** AI 串联的是节点和边，不是豪华编辑器。四类节点全部在 P0 就作为一等公民存在、可互相连边；富编辑器按需后置。这是保住排期的那把刀。
- [x] 反馈收件箱：证据节点列表（创建/编辑/删除）
- [ ] 文档页富编辑器：TipTap/BlockNote（**0.3 单独 spike，当前用纯文本域**）
- [x] 看板：任务节点按 `board_status` 列拖拽（`@dnd-kit`）
- [x] **结果节点：手动录入指标快照**——P0 已纳入
- [x] 手动连边 UI：8 种边类型下拉 + `NodePicker`（选已有/即时新建目标节点）+ 可选出处备注
- [x] 节点详情页：正文/关联边/出处三 tab，替代"侧栏"设计（更适合 Phase 0 桌面布局）

### 🚪 P0 出口验收 —— ✅ 通过 2026-07-05
- [x] 纯手动完成：录 3 条证据 → 建 1 个需求（挂支撑边）→ 拆 2 张任务卡（挂实现边）→ 录 1 条结果（挂验证边）——通过 `lib/graph` 脚本构建，浏览器逐页 snapshot 视觉确认渲染正确
- [x] 任一节点修改即时反映到所有视图（同一数据源）——Server Component 直读 Postgres + `revalidatePath`，无客户端缓存漂移风险
- ⚠️ **验证方式说明**：浏览器自动化点击在本次 preview 环境中有工具本身的限制（未能触发 React 19 的 Server Action 表单拦截，根因排查见 Phase0-开工计划.md 0.2 小节）；改用 curl 直接验证 Next.js 表单机制 + vitest 覆盖新增函数 + 脚本驱动完整链路 + 浏览器视觉核对，作为等效替代验证

## Phase 1：Agent 运行时 + 捕获 + 审批 + 侧边栏（PRD R3、R5、R6）

> 开工计划详见《Phase1-开工计划.md》（含三个岔路口定案：pg-boss 异步薄编排 / 只做粘贴+文件入口 / 低风险边写入即 confirmed 可撤销）。

**目标**：录入这件脏活从此归 AI；人只审不录。

### 1.1 Agent 运行时（架构 5.2/5.3）—— ✅ 完成
- [x] 事件层：Postgres LISTEN/NOTIFY（只服务 SSE 实时推送）+ pg-boss 任务队列
- [x] 统一五段流水线骨架：触发→上下文装配→LLM（tool-calling，见 0.4 spike 结论，不是 `generateObject`）→zod 校验→写提议
- [x] 四份合同的配置结构（触发/权限/schema/模型）——先只实现捕获合同
- [x] 数据库双凭证：读写凭证（捕获用）与只读凭证（`graph_readonly` 角色，检索用，物理隔离）

### 1.2 捕获 Agent（R3）—— ✅ 完成

- [x] 出处自动填充：每个提议节点/边带原始输入片段引用 + 置信度 + 批次键

### 1.3 审批（R5）—— ✅ 完成

- [x] 接受/拒绝日志按边类型入库（v2 放权的原料，只记不用）

### 1.4 AI 侧边栏（R6）—— ✅ 完成

- [x] 锚定机制：跟随当前节点/页面，<1s 切换上下文
- [x] 捕获入口 + 审批 tab 收纳进侧边栏
- [x] 关闭态验证：无任何 AI 元素残留，P0 全功能可用

### 1.5 提前的占位 surface（保证 P2 串联 demo 能横跨全部五个面）—— ✅ 完成

## Phase 2：检索/追溯 + 冷启动导入（PRD R4、R7）

**目标**：产品的"哇时刻"上线——一键问出"当初为什么"。

### 2.1 语义索引
- [ ] Embedding 管线：节点创建/更新时自动向量化（pgvector）
- [ ] Project scope 过滤 + 升级式检索（局部无果→查全局并标注来源项目）

### 2.2 检索 Agent + 追溯搜索栏（R4）
- [ ] 两段式检索：向量定位入口 → 沿类型边遍历（纯 SQL，零幻觉）
- [ ] 生成层硬约束：仅可引用遍历到的节点；每句话挂深链；无引用陈述直接拦截
- [ ] "图里没有记录"路径（明确拒答，不推测）
- [ ] Cmd-K 全局追溯搜索栏
- [ ] 检索评估集 v1：≥20 个问答对（含 5 个"应拒答"用例），CI 跑精确率

> **交互风格备忘（2026-07-06 讨论）**：答案应该是自然语言段落，引用以行内蓝色链接嵌入句子中（点开直达节点），不是列表/卡片式的"来源"区块——读起来应该像正常对话里引一句话，不是表单。P1 侧边栏"上下文"面板目前是结构化的边列表（"支撑 → xxx"），P2 检索栏做完后，一并回头看要不要把"上下文"面板也改写成同样的自然语句+行内链接风格，到时候整体感受一遍再决定。

### 2.3 冷启动导入（R7）
- [ ] 通用 Markdown/CSV 导入 → 走捕获管线熔图
- [ ] Notion 导入器（导出包解析 → 预处理头）
- [ ] 飞书导入器
- [ ] 大批量并行：同一运行时多 worker（5000 条级别压测）

### 🚪 P2 出口验收 = Demo 故事线完整可演
- [ ] ① 导入真实历史（Notion/飞书导出）→ ② 审批确认因果边（5 分钟）→ ③ Cmd-K 提问"X 当初为什么"→ ④ 回答每句可点击直达节点 → ⑤ 关侧边栏，工作台完整可用
- [ ] **串联 demo**：从一个需求节点出发，一路点通 证据链 → PRD → 原型占位 → 任务 → 结果，五个 surface 全部由边连起、可互相跳转（验证核心 usecase）
- [ ] 检索评估集精确率达标（错误引用 0 容忍）
